## 0. このドキュメントの読み方（AIエージェント向け）

このドキュメントは、AIコーディングエージェントが**自律的にコード修正を行うための作業仕様書**である。各セクションは以下の構造を持つ。

- **目的**：何のための変更か
- **対象ファイル / 影響範囲**：触ってよい場所と触ってはいけない場所
- **実装手順**：ステップごとの具体的な作業
- **完了条件（受入基準 / DoD）**：客観的に測定可能な合格ライン
- **ロールバック手順**：失敗時に元に戻す方法
- **テスト手順**：作業後に必ず実行すべき検証

AIエージェントは、**1セクション完了するごとに必ず「完了条件」と「テスト手順」を実行し、結果をレポートしてから次に進むこと**。複数セクションをまとめて実装してはならない。

## 1. 背景と現状の問題

クライアント（SEO計測担当）から、3回計測のデータと共に以下の課題が提示されている。

### 1-1. サーバー応答が遅い（最優先）

3回計測の生データは以下のとおり。

| 計測項目 | 値 |
| --- | --- |
| HTML取得時間（3回平均） | 約 4.3 秒（3,224 / 3,513 / 6,079 ms） |
| X-Vercel-Cache | MISS（キャッシュされていない） |
| Cache-Control | `private, no-cache, no-store, max-age=0, must-revalidate` |
| X-Vercel-Id | `hnd1::iad1::…`（東京エッジ受信 → 米国 iad1 でレンダリング） |

**根本原因：**

1. レスポンスヘッダで明示的に `no-store` を返しており、CDNにもブラウザにも一切キャッシュされていない。福岡店ページなど更新頻度が低い静的寄りの内容にも関わらず、毎リクエスト毎にフルSSRが走っている。
2. レンダリングリージョンが `iad1`（米国東部バージニア）。日本のユーザーには太平洋を往復するレイテンシが上乗せされている。

### 1-2. 画像が重い・最適化されていない

| 計測項目 | 値 |
| --- | --- |
| ページ内画像 | 35ユニーク、実画像合計 約 9.1 MB |
| 配信元 | すべて Supabase Storage から無加工で直接配信（`<img src="…supabase.co/storage/…">`） |
| Next.js `<Image>` の使用 | 0件（`/_next/image` 経由が一切ない） |

**サイズが特に大きいファイル（要圧縮・リサイズ）：**

| サイズ | ファイル |
| --- | --- |
| 2,054 KB | `gallery/…/1776677442671_wszuxy.webp` |
| 1,499 KB | `news-pages/…/1774079527403.png`（PNGのまま） |
| 1,499 KB | `news-pages/…/1772542100643.jpg` |
| 580 KB | `news-pages/…/1772542100643.jpg`（別） |
| 562 KB | `store-top/fukuoka/header_1772497484207.png` |
| 493 KB | `store-top/fukuoka/footer_1774070825047.png` |
| 473 KB | `store-top/fukuoka/flow_1775799580965.webp` |
| 374 KB | `store-top/fukuoka/footer_1774070971715.png` |

### 1-3. その他の確認結果（現状で問題なし／参考情報）

これらは**変更不要**。誤って触らないこと。

| 項目 | 状況 |
| --- | --- |

## 2. 改善の到達目標（KGI / KPI）

### 2-1. KGI（最終目標）

- **HTML 取得時間（TTFB）を 3回平均 1.0 秒以下にする**（現状 4.3 秒 → 約 75% 短縮）
- **ページ内画像合計サイズを 3.0 MB 以下にする**（現状 9.1 MB → 約 67% 削減）

### 2-2. KPI（中間指標）

| 指標 | 目標値 | 計測方法 |
| --- | --- | --- |
| `X-Vercel-Cache` | `HIT`（2回目以降） | `curl -I` のレスポンスヘッダ |
| `X-Vercel-Id` のリージョン | `hnd1::hnd1::…`（東京で完結） | `curl -I` のレスポンスヘッダ |
| 1 ファイル当たりの最大画像サイズ | 500 KB 以下 | Chrome DevTools Network タブ |
| LCP（Largest Contentful Paint） | 2.5 秒以下 | PageSpeed Insights |
| CLS（Cumulative Layout Shift） | 0.1 以下（現状から悪化させない） | PageSpeed Insights |
| Lighthouse Performance スコア | モバイルで 80 以上 | Chrome Lighthouse |

## 3. 絶対に守るべき制約（最重要）

AIエージェントは以下の制約を**いかなる場合も破ってはならない**。違反した場合は即座に作業を停止し、人間にエスカレーションすること。

### 3-1. 既存機能を壊さないための禁止事項

| カテゴリ | 禁止内容 |
| --- | --- |
| デザイン | 既存のレイアウト、配色、フォント、アイコン、余白の変更 |
| 文言 | 既存のテキストコンテンツ、メタタグ、構造化データの改変 |
| URL | 既存のページURL（`/store/[slug]`、`/news/[id]` 等）の変更・リダイレクト追加 |
| API契約 | Supabase テーブル定義、RPC、エッジ関数の入出力スキーマの変更 |
| 認証 | 既存の認証フロー、Cookie 名、セッション仕様の変更 |
| 環境変数 | 既存の環境変数名の変更（追加は可、リネーム・削除は不可） |
| 依存パッケージ | 既存パッケージのメジャーバージョン更新（マイナー/パッチは可） |
| データベース | スキーマの破壊的変更（カラム削除・型変更） |

### 3-2. 必須の安全策

1. **すべての変更は別ブランチで行う**（`feat/perf-improvement` 等）。`main` への直接コミット禁止。
2. **1セクションごとに必ずコミットを分ける**。「画像最適化」と「キャッシュ設定」を1コミットにまとめてはならない。
3. **デプロイは Vercel Preview で必ず動作確認**。Production への即時マージ禁止。
4. **既存の E2E / ユニットテストが存在する場合、変更後に全件パスすること**。失敗したら作業を止める。
5. **ユーザー認証が必要なページのキャッシュ設定は変更してはならない**（後述）。
6. **既存の画像 URL が動的に組み立てられている箇所**（Supabase Storage から取得する `getPublicUrl()` 等）の戻り値構造は維持する。

### タスク 4-1【最優先】`/store/[slug]` の HTML を CDN キャッシュ可能にする

#### 目的

`Cache-Control: no-store` を撤廃し、Vercel Edge CDN にキャッシュさせる。これだけで TTFB が劇的に改善する見込み。

#### 対象ファイル / 影響範囲

- **対象**：`app/store/[slug]/page.tsx`（または `pages/store/[slug].tsx` の Pages Router 構成の場合）
- **対象**：`middleware.ts`（あれば、no-store を上書きしている可能性がある）
- **対象**：`next.config.js` の `headers()` 設定
- **対象外**：会員ページ、認証必須ページ、フォーム送信後の動的ページ（これらは `no-store` のまま維持）

#### 実装手順

1. **現状調査**
   - `git grep -n "no-store"` で `no-store` を出している全箇所を洗い出す。
   - `app/store/[slug]/page.tsx` のキャッシュ関連設定（`export const dynamic`、`export const revalidate`、`fetch(url, { cache: ... })` 等）を確認する。
   - `middleware.ts` で `Cache-Control` を書き換えていないか確認する。
   - Supabase クライアントの `createServerClient` で Cookie ベースのセッション処理がある場合、それが SSR を強制している可能性を確認する。

2. **方針の選択（以下のいずれか1つを採用）**

   **方針A（推奨）：ISR（Incremental Static Regeneration）**
   ```tsx
   // app/store/[slug]/page.tsx の先頭に追記
   export const revalidate = 300; // 5分ごとに再生成
   export const dynamic = 'force-static';
   ```

   **方針B：Cache-Control ヘッダの明示**
   ```ts
   // next.config.js
   async headers() {
     return [
       {
         source: '/store/:slug',
         headers: [
           {
             key: 'Cache-Control',
             value: 'public, s-maxage=300, stale-while-revalidate=600',
           },
         ],
       },
     ];
   }
   ```

   **方針C：`unstable_cache` を使用（部分的にキャッシュ）**
   ```ts
   import { unstable_cache } from 'next/cache';
   const getStoreData = unstable_cache(
     async (slug) => { /* Supabase fetch */ },
     ['store-data'],
     { revalidate: 300, tags: [`store-${slug}`] }
   );
   ```

3. **認証関連の切り分け**
   - `/store/[slug]` ページがログイン状態によって表示内容が変わる場合（例：お気に入りボタン、レビュー投稿フォームなど）、その部分のみ Client Component に切り出して、ページ本体は静的化する。
   - ヘッダーのログイン状態表示は **Client Component + `useSession()` で取得**するよう変更。ページ本体の SSR でログイン情報を参照しないこと。

4. **revalidateTag による更新フック**
   - 管理画面で店舗情報を更新した際、`revalidateTag(`store-${slug}`)` を呼び出すように API Route を追加。これにより、キャッシュ期限前でも即座に反映可能。

#### 完了条件（DoD）

```bash
# 以下を満たすこと
curl -I https://[本番URL]/store/fukuoka
# 1回目: X-Vercel-Cache: MISS（または STALE）
# 2回目以降: X-Vercel-Cache: HIT
# Cache-Control に public, s-maxage が含まれている
```

- 3回計測の平均 TTFB が 1.0 秒以下。
- ログイン状態によって出し分けるはずの UI（お気に入りマーク等）が正しく機能している。

#### ロールバック手順

```bash
git revert <commit-hash>
git push origin feat/perf-improvement
# Vercel で即座に Preview デプロイされる
```

万一 Production に出してしまった場合、Vercel ダッシュボードから「Promote to Production」で 1つ前のデプロイに即座に戻す。

#### テスト手順

1. `npm run build` がエラーなく完了すること。
2. `npm run dev` でローカル起動し、`/store/fukuoka` 等の代表ページが正しく表示されること。
3. Vercel Preview デプロイ後、`curl -I` で `X-Vercel-Cache: HIT` を確認。
4. ブラウザで「シークレットモード」「通常モード」「ログイン後」の3パターンで表示確認。
5. Lighthouse でモバイル Performance スコアを計測し、改善前後で比較。

### タスク 4-2 Vercel の Function Region を東京（`hnd1`）に固定

#### 目的

`iad1`（米国東部）でのレンダリングを `hnd1`（東京）に変更し、日本のユーザーとの物理的距離を縮める。

#### 対象ファイル / 影響範囲

- **対象**：`vercel.json`（プロジェクトルート）
- **対象**：各 Route Handler / Server Action の `runtime` 設定
- **対象外**：Supabase の接続先（既に東京リージョンに配置されているはず。確認は必要）

#### 実装手順

1. **現状確認**
   - `vercel.json` が存在するか確認。なければ新規作成。
   - Vercel ダッシュボード → Settings → Functions → Region で現在の設定を確認。

2. **設定追加**
   ```json
   {
     "regions": ["hnd1"]
   }
   ```
   または、Pro プラン以上の場合は Vercel ダッシュボードで GUI から設定可能。

3. **Edge Runtime と Node.js Runtime の切り分け**
   - 軽量なルートは `export const runtime = 'edge'` でエッジ実行を検討（ただし Supabase の Node.js 専用 API を使う場合は不可）。
   - Supabase との通信を行うルートは Node.js Runtime のまま、リージョンのみ `hnd1` に固定。

4. **Supabase リージョンの確認**
   - Supabase ダッシュボード → Settings → General → Region が「Tokyo (ap-northeast-1)」になっていることを確認。
   - 異なる場合は、別途プロジェクトオーナーに相談（マイグレーションが必要なため、本タスクの範囲外）。

#### 完了条件（DoD）

```bash
curl -I https://[本番URL]/store/fukuoka
# X-Vercel-Id: hnd1::hnd1::xxxxx-xxxxx
# （受信エッジもレンダリングリージョンも hnd1 になっている）
```

#### ロールバック手順

`vercel.json` の `regions` フィールドを削除して再デプロイ。

#### テスト手順

1. デプロイ後、`curl -I` で X-Vercel-Id を確認。
2. 日本国内から複数回アクセスし、TTFB が安定して短縮されていること。

### タスク 4-3 画像配信の最適化（`next/image` または Supabase Image Transform 経由）

#### 目的

Supabase Storage からの直配信をやめ、リサイズ・WebP/AVIF 変換・適切な `srcset` を経由した配信に切り替える。特に 1MB 超のファイルを優先的に対応する。

#### 対象ファイル / 影響範囲

- **対象**：`<img src="…supabase.co/storage/…">` を使っている全箇所
- **対象**：`next.config.js` の `images.remotePatterns` 設定
- **対象外**：既に `<Image>` を使っている箇所（あれば触らない）
- **対象外**：ロゴ・アイコン等の SVG（既に最適化済みのため）

#### 実装手順

1. **現状の画像使用箇所の洗い出し**
   ```bash
   git grep -n "supabase.co/storage" -- '*.tsx' '*.jsx' '*.ts' '*.js'
   ```
   結果を一覧化し、各画像の使われ方（固定サイズ／可変サイズ／背景画像）を分類する。

2. **方針の選択**

   **方針A（推奨）：`next/image` への置き換え**
   ```tsx
   // 変更前
   <img src={`https://xxxx.supabase.co/storage/v1/object/public/news/${file}`} alt="…" />

   // 変更後
   import Image from 'next/image';
   <Image
     src={`https://xxxx.supabase.co/storage/v1/object/public/news/${file}`}
     alt="…"
     width={800}
     height={600}
     sizes="(max-width: 768px) 100vw, 800px"
     // 重要画像のみ priority、それ以外は lazy（デフォルト）
   />
   ```

   `next.config.js`:
   ```js
   module.exports = {
     images: {
       remotePatterns: [
         { protocol: 'https', hostname: '*.supabase.co', pathname: '/storage/**' },
       ],
       formats: ['image/avif', 'image/webp'],
     },
   };
   ```

   **方針B：Supabase Image Transformation を使う**
   ```tsx
   const url = supabase.storage.from('news').getPublicUrl(file, {
     transform: { width: 800, quality: 80, resize: 'contain' },
   }).data.publicUrl;
   <img src={url} alt="…" loading="lazy" />
   ```

   **推奨は方針A**（`next/image`）。理由：自動 `srcset`、AVIF 変換、Lazy Loading、レイアウトシフト防止（`width`/`height` 必須）がすべて含まれるため。

3. **大容量ファイルの優先対応リスト（最低限これだけは対応）**

   | 優先度 | ファイル | 元サイズ | 対応 |
   | --- | --- | --- | --- |
   | P0 | `gallery/.../1776677442671_wszuxy.webp` | 2,054 KB | リサイズ（表示用途用の解像度に） |
   | P0 | `news-pages/.../1774079527403.png` | 1,499 KB | WebP/AVIF に変換 |
   | P0 | `news-pages/.../1774079527403.jpg` | 1,499 KB | リサイズ＋WebP変換 |
   | P1 | `news-pages/.../1772542100643.jpg` | 580 KB | リサイズ＋WebP変換 |
   | P1 | `store-top/fukuoka/header_1772497484207.png` | 562 KB | WebP/AVIFに変換 |
   | P1 | `store-top/fukuoka/footer_1774070825047.png` | 493 KB | WebP/AVIFに変換 |
   | P2 | `store-top/fukuoka/flow_1775799580965.webp` | 473 KB | リサイズ |
   | P2 | `store-top/fukuoka/footer_1774070971715.png` | 374 KB | WebP/AVIFに変換 |

4. **`width` / `height` の決定方針**
   - PC：実際の表示サイズの 1.5 倍まで（Retina 対応）
   - モバイル：実際の表示サイズの 2.0 倍まで
   - `sizes` 属性で `(max-width: 768px) 100vw, 800px` のようにレスポンシブ対応

5. **`fetchpriority="high"` の維持**
   - 既存で 5 枚に `fetchpriority="high"` が設定されている（ファーストビュー画像）。これらは `next/image` の `priority` プロパティに置き換えること。

#### 完了条件（DoD）

- Chrome DevTools Network タブで、画像 1 枚当たり 500 KB 以下になっていること。
- ページ全体の画像合計サイズが 3 MB 以下。
- 各画像が `/_next/image?url=…` 経由（または Supabase Transform 経由）で配信されている。
- `width` / `height` が必ず指定されており、CLS が悪化していない。
- 画像が表示崩れを起こしていない（PC / モバイル / タブレット）。

#### ロールバック手順

ファイル単位でコミットを分けておけば、画像が壊れたページのみ `git revert` で戻せる。

#### テスト手順

1. ローカルで `npm run build && npm run start` で起動し、代表的な画像が表示されることを確認。
2. Vercel Preview で実機 / シミュレーターで表示確認（PC / モバイル / iPad）。
3. Chrome DevTools の Network タブで画像サイズを計測。
4. Lighthouse で LCP / CLS を計測。

### タスク 4-4 アップロード時の自動圧縮・最大幅制限を実装

#### 目的

`news-pages` / `gallery` に画像がアップロードされる際、自動的に圧縮・リサイズして、Supabase Storage に肥大化したファイルが入らないようにする。

#### 対象ファイル / 影響範囲

- **対象**：管理画面の画像アップロード処理（`app/admin/news/upload` 等、要調査）
- **対象**：Supabase Storage への `upload()` 呼び出し箇所
- **対象外**：既にアップロード済みの過去ファイル（タスク 4-3 でカバー）

#### 実装手順

1. **アップロード処理の特定**
   ```bash
   git grep -n "supabase.storage" -- '*.tsx' '*.ts' '*.js' | grep -i upload
   ```

2. **クライアント側で圧縮**

   ライブラリ候補：`browser-image-compression`
   ```ts
   import imageCompression from 'browser-image-compression';

   async function handleUpload(file: File) {
     const compressedFile = await imageCompression(file, {
       maxSizeMB: 0.5,        // 最大 500 KB
       maxWidthOrHeight: 1920, // 最大幅 1920px
       useWebWorker: true,
       fileType: 'image/webp', // WebP に変換
     });
     await supabase.storage.from('news-pages').upload(path, compressedFile);
   }
   ```

3. **サーバー側で検証（ダブルチェック）**

   Supabase Edge Function でアップロード時にファイルサイズを検証し、超過していたら拒否。
   ```ts
   if (file.size > 1024 * 1024) {
     throw new Error('画像サイズは 1MB 以下にしてください');
   }
   ```

4. **管理者向けの UI フィードバック**
   - 圧縮中・圧縮後のサイズを表示。
   - 元ファイルサイズと圧縮後サイズの比較。

#### 完了条件（DoD）

- 5 MB の画像をアップロードしても、Supabase Storage には 500 KB 以下のファイルが保存される。
- WebP 形式で保存されている。
- 既存のアップロード機能が動作している（投稿、サムネイル表示等）。

#### ロールバック手順

該当コミットを `git revert`。圧縮処理を外しても、既存のアップロードロジック自体は維持しているため安全。

#### テスト手順

1. 大容量画像（5 MB JPG、3 MB PNG 等）でアップロードテスト。
2. Supabase Storage 上のファイルサイズを確認。
3. アップロード後の表示確認。

## 6. リスクと緩和策

| リスク | 影響 | 緩和策 |
| --- | --- | --- |
| キャッシュ化により古いデータが表示される | 店舗情報の更新が反映されない | `revalidateTag` を使い、更新時に明示的にキャッシュをパージする |
| 認証ページがキャッシュされ、他人のセッションが表示される | 個人情報漏洩（重大） | 認証必須ページは `dynamic = 'force-dynamic'` を明示。Middleware で `/mypage/*` 等を除外 |
| 画像 URL の変更でリンク切れ | 過去の SNS 投稿等から飛んできた画像URLが 404 | Supabase Storage の元 URL は維持し、ページ内では `next/image` 経由に変更するのみ。直リンクの URL は変えない |
| `next/image` の最適化サーバーが Vercel の Image Optimization の枠を超過 | 課金発生 | Vercel ダッシュボードで使用量を監視。月間 5,000 ソース画像が無料枠 |
| Function Region 変更で Supabase との通信が遅くなる | DB クエリのレイテンシ増 | Supabase のリージョンと一致していることを事前確認（Tokyo / ap-northeast-1） |
| 画像圧縮で画質が劣化しすぎる | UI の見栄え悪化 | `quality: 80` をデフォルトとし、重要画像（ファーストビュー）は `quality: 90` |

## 7. AIエージェント運用ルール

### 7-1. エージェント開始時の必須確認

AIエージェントは作業開始前に、以下を必ず実行すること。

```bash
# 1. 現在のブランチを確認
git branch --show-current
# main 以外のブランチで作業すること

# 2. 作業前の状態をタグで保存
git tag pre-perf-work-$(date +%Y%m%d-%H%M%S)
git push --tags

# 3. 依存パッケージのバージョンを確認
cat package.json | grep -E "(next|react|@supabase)"

# 4. ビルドが通る状態か確認
npm run build
```

### 7-2. エージェント完了時の必須レポート

各タスク完了時、以下のフォーマットでレポートを出すこと。

```markdown
## タスク完了レポート：[タスク番号 / 名称]

### 実施した変更
- 変更ファイル：(リスト)
- 追加した行数 / 削除した行数

### 計測結果
- 変更前 TTFB: ___ ms（3回平均）
- 変更後 TTFB: ___ ms（3回平均）
- 変更前 画像サイズ: ___ MB
- 変更後 画像サイズ: ___ MB

### テスト結果
- `npm run build`: ✅ / ❌
- `npm run lint`: ✅ / ❌
- 既存テスト: ✅ / ❌
- 目視確認（5ページ）: ✅ / ❌

### 懸念事項 / エスカレーション
(あれば記述)
```

### 7-3. 作業を停止すべき状況

以下のいずれかが発生した場合、AIエージェントは即座に作業を停止し、人間にエスカレーションすること。

1. `npm run build` が失敗する状態が 3 回以上続いた。
2. 既存のテストが失敗する。
3. 「絶対に守るべき制約（第 3 章）」に該当する変更が必要だと判断した。
4. Supabase / Vercel のリージョン設定が不明、または変更権限がない。
5. Cache-Control を変更しようとしているページに、認証関連の処理が含まれていた。
6. 画像配信先のドメインが Supabase 以外にも存在することが判明した（CDN 移行があった可能性）。
