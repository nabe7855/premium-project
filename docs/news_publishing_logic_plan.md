# ニュース記事の「各店舗独立ステータス」および「公開日時」ロジック修正計画

クライアントからの要望に基づき、1つの記事を複数店舗に紐づけた際、**「店舗ごとの公開・非公開」と「店舗ごとの公開日時」を完全に独立させる**ための修正プランを策定しました。

## 1. 現状の課題
現在、データベース（`PageRequest`テーブル）は以下のような構造になっています。
* 全体で1つの `status`（公開/非公開）を持っている
* 全体で1つの更新日時（`updatedAt`）を持っている
* 対象店舗は `targetStoreSlugs` (例: `["yokohama", "fukuoka"]`) という単なる配列で管理している

そのため、「福岡店だけ非公開にしたい」「横浜店だけ今日の日付で再公開したい」といった個別の操作ができず、どの店舗で操作しても記事全体のステータスが連動して変わってしまいます。

---

## 2. データ構造の変更プラン
この問題を解決するため、各店舗ごとの「ステータス」と「公開日時」を保持する構造に変更します。

### プラントライアル（JSONフィールドの活用）
既存の `referenceUrls`（追加データ用のJSONカラム）の中に、各店舗の独立した設定値を持たせます。これなら大規模なデータベース移行（マイグレーション）をせずとも柔軟に実装可能です。

**【データ構造のイメージ】**
```json
// referenceUrls の中身
{
  "storeSettings": {
    "yokohama": {
      "status": "published",
      "publishedAt": "2026-03-21T10:00:00Z"
    },
    "fukuoka": {
      "status": "private",
      "publishedAt": null
    }
  }
}
```
※既存の全体 `status` カラムは「管理者用の下書き判別」などに残すか、使用を停止します。

---

## 3. 管理画面 (UI) の変更
1. **「全体公開」ボタンの廃止**
   現在の記事作成画面にある「公開/非公開」の単一スイッチを廃止（または一括操作ボタンに変更）します。
2. **個別の店舗ごとのスイッチと日付設定**
   「配信先店舗」を選んだあと、それぞれの店舗（例：横浜店・福岡店）の横に、独立した**「公開スイッチ」**と**「公開日時のカレンダー入力エリア」**を表示します。

---

## 4. 表示側のロジック変更（バックエンド/フロントエンド）
* **一覧の取得処理**（`getPublishedPagesByStore` など）
  全体ステータスではなく、`storeSettings[対象の店舗slug].status === 'published'` かどうかを見て公開記事を絞り込みます。
* **並べ替えロジック**
  `updatedAt` ではなく、各店舗ごとの `storeSettings[対象の店舗slug].publishedAt` を基準に新しい日付順で並び替えます。これにより、特定店舗だけの「最新情報への浮上」なども正確に制御できます。

---

こちらのプランで進めることで、ご提示いただいた「他の店舗設定に影響を及ぼさずに、指定した店舗だけ公開/非公開や公開日をコントロールする」機能がきれいに実現できます！問題なければ、この仕様に沿ってソースコードやUIの改修作業に入ります。
